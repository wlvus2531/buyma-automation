### v3.1의 치명적 문제 (코드 분석에서 발견)

1. **가격·마진이 전부 LLM 환각 추정치** — `sourcing-engine`이 Claude에게 cost_krw/list_price_jpy/margin_pct를 "추정"으로 생성시킴. 실측 근거 0. 이 값이 승인 룰의 입력이 됨 → 의사결정 체계 전체가 허수 위에 있음
2. **소싱 근거 부재** — ai_score는 프롬프트 텍스트 기준의 자의적 점수. 바이마 내 실제 수요 데이터(찜/조회/등록일/경쟁 셀러)와 무관
3. CS cron 단절 (`runDailyCSBatch`가 어디서도 호출 안 됨)
4. 핵심 테이블 CREATE 마이그레이션이 저장소에 없음 (스키마 버전관리 밖)
5. RLS가 anon에 전부 허용 + CRON_SECRET 미설정 시 cron 공개
6. 네이버 첫 검색결과 무검증 매칭 → 엉뚱한 상품/가격
7. 매일 순수 insert → 중복 상품 누적

## 1. v4 핵심 설계 원칙

**"바이마 실측 데이터가 소싱의 입력이다"**

v3.1: Claude 상상 → 상품 생성 → 네이버로 뒷받침 시도
v4: **바이마 크롤링(실수요 신호) → 후보 추출 → 국내 구매처 검증(실가격) → 마진 계산(실계산) → AI는 분석·요약·파생만 담당**

AI의 역할 변경: "상품을 지어내는 존재" → "수집된 실측 데이터를 분석해 근거를 요약하고 파생 아이디어를 내는 존재"

### 데이터 수집 아키텍처 (기술적 핵심 결정)

바이마는 서버사이드(Vercel) 크롤링을 차단할 가능성이 높음. 따라서:

- **Chrome 확장 = 리서치 수집기 (핵심 인프라로 승격)**
  - 대표(또는 상시 켜둔 PC)의 브라우저에서 바이마 페이지를 순회하며 데이터 수집 → API로 전송
  - v3.1의 monitor content.js가 이미 이 패턴을 증명함 (JSON-LD + DOM 파싱 → `/api/monitor/report`)
  - v4에서는 "수집 미션" 개념 도입: 서버가 오늘의 리서치 미션(입구 URL 목록)을 생성 → 확장이 순차 방문·수집 → 서버가 분석
- 보조: Vercel 서버에서 가능한 것(공개 JSON-LD, 이미지 URL 패턴)은 서버에서, 로그인/동적 데이터는 확장에서

## 3. 모듈 1 — 소싱기 v4 (최우선)

### 3-1. DB 신규 테이블

```
research_missions   — 오늘의 리서치 미션 (입구 URL, 방법 A/B/C, 상태)
buyma_candidates    — 바이마에서 수집한 후보 상품 (원천 데이터)
  · buyma_item_id, url, name_jp, brand, price_jpy, wish_count(찜), access_count(조회),
    inquiry_recent(최근문의 여부), listed_date(썸네일 URL 날짜 추출), seller_id,
    seller_type(일반/프리미엄/대량), method(A|B|C), mission_id, raw jsonb
sellers_watch       — 모델링 대상 셀러 (방법 B) — 팔로워/출품수/평점/주문실적 추적
purchase_sources    — 상품별 구매처 후보 (검증된 실가격)
  · product_id, mall, url, price_krw, is_whitelisted, checked_at
brand_rules         — 하드 필터 (금지/제한/이미지주의 브랜드 + 금지/신뢰 사이트) ★코드가 아닌 DB
```

### 3-2. 방법 A (인기상품 리서치) 자동화 분해

| 강의 단계 | v4 구현 |
|---|---|
| 입구 선정 (매일 다른 입구) | `research_missions` 생성기: 카테고리 로테이션 + 최근 성과 가중치 |
| '한국' 필터 → 노브랜드 제거 → 인기순 | 확장이 필터 조합된 URL 방문 |
| 1~2페이지 초인기 제외, 프리미엄 쇼퍼 제외 | 수집 후 서버 필터 (rank > N, seller_type 판별) |
| 조회 100+/찜 3+ · 최근 문의 · 등록 2~3개월 내 | 상품 페이지 수집 + **썸네일 URL 날짜 추출** (검증 항목 V1) |
| 셀러 주문실적에서 최근 판매 확인 | 셀러 페이지 수집 (검증 항목 V2) |
| 구글링으로 구매처·구매가 확인 | 네이버 API + 화이트리스트 편집샵 검색 → `purchase_sources` |
| 부가세 환급 전/후 마진 5%+ | **마진 계산기 모듈** (올인원 시트 로직 이식 + 환급 계산 추가) |

### 3-3. 방법 B (모델링 리서치)

- `sellers_watch`에 모델링 셀러 등록 (초기: 수동 발굴 + AI 후보 추천)
- 발굴 조건 구현: 팔로워 대비 출품수 높음 + 평점/주문실적 → 확장이 셀러 페이지 주기 수집
- 해당 셀러 신규 출품 감지 → 자동으로 방법 A와 동일한 검증 파이프라인에 투입

### 3-4. 방법 C (파생 리서치) — AI의 진짜 역할

판매 확정/인기 확인된 상품에 대해 Claude가 파생리서치 3-STEP 수행:
1. 특징 키워드 10개 추출
2. "왜 팔리는가" 언어화
3. 유사 특징 파생 상품 후보 생성 → 확장 수집 미션으로 재투입 (선순환)

### 3-5. 하드 필터 (코드 레벨 강제)

- `brand_rules` 테이블 시드: 지재권 58 + 출품제한 7 + 판매권한 필요 60여 + 금지 구매처 4 + 화이트리스트 12
- 파이프라인 모든 단계 입구에서 차단 (수집 시, 셀렉 시, 등록 시 3중)
- 이미지 주의 브랜드는 차단이 아닌 "직접 촬영/후기 이미지 필수" 플래그

## 4. 모듈 2 — 근거 기반 셀렉

기존 `/products` 큐 모드 유지 + **근거 카드**로 전면 개편:

```
[썸네일]  상품명 / 브랜드
근거: 찜 12 · 조회 340 · 등록 47일차 · 최근문의 3일 전
경쟁: 셀러 4명 · 최저 ¥6,800 · 우리 예상가 ¥6,400
마진: 구매가 ₩41,650 (29CM ✓화이트리스트) → 환급 전 7.2% / 환급 후 12.8%
발굴: 방법 A · 입구 "레이디스 모자 > 캡"
[👍 선택 (Y)]  [👎 패스 (N)]
```

관리 시트 (업로드 확정 리스트): 소싱날짜 · 업로드날짜 · 상품명 · 판매가 · 구입예상가 · 구입처URL · 마진율(환급 전/후) · 판매여부 · 판매수량 — Supabase 테이블 + `/sourcing-sheet` 페이지. 필터/정렬/CSV 내보내기.

## 5. 모듈 3 — 업로드 준비

1. **상품명 생성기**: 수집된 경쟁 셀러 상품명(원천 데이터 보유!) 참고 + 키워드 풀(★関税込★ 등 10종) 조합 — Claude 생성 + 금지 표현 검사
2. **썸네일 템플릿**: 코드 기반 합성(`sharp` 또는 `canvas`) — 배경 제거/통일 프레임/브랜드 텍스트 배치 템플릿. 모델 얼굴 감지 시 리젝 (검증 항목 V4). 프로그램 내 다운로드 버튼
3. **상세페이지**: 코멘트+색사이즈 템플릿(확보됨)을 `listing_templates` 테이블에 저장, 상품별 가운데 설명 부분만 AI 작성

## 6. 모듈 4 — 가격 방어 (모니터링 리뉴얼)

- 기존 확장 monitor 유지·확장: 매일 2회 우리 출품 상품 페이지 순회
- 최저가 아니면: `신가격 = max(경쟁최저가 - 1엔, 마진하한가)` 자동 계산
- 마진하한가 = 모듈 2의 실측 원가 기반 (환급 후 마진 5% 방어선)
- 가격 변경 실행: 확장 autofill 패턴으로 바이마 가격 수정 폼 자동 입력 (검증 항목 V3) — 초기엔 "제안 → 사장님 원클릭 승인 → 확장 실행"

## 7. 모듈 5 — 주문관리

- `orders` 테이블 정식화 (현 인메모리 제거): 올인원 시트의 주문관리 컬럼 이식 (주문일/번호/상품/판매가/발주여부/구매금액/카드/배대지입고/발송일/운송장/정산액/실마진/상태)
- 주문 감지: 바이마 주문관리 페이지를 확장이 수집 (엑셀 다운로드 가능 여부는 검증 항목 V5)
- **구매처 재탐색**: 주문 발생 시 `purchase_sources` 재검증 + 신규 최저가 탐색 → 추천
- **조광 배대지 엑셀**: 지정 형식 자동 생성 (형식 파일은 대표에게 요청 — 결정 필요 D3)
- **운송장 자동 등록**: 상용 확장(BUYJW2604)이 이미 해결 → 재구현하지 않고 병행 사용, 등록 결과만 `orders`에 반영하는 가벼운 연동 (CSV 이력 임포트)

## 8. 모듈 6 — CS 관리

- Gmail 연동(MCP가 아닌 앱 레벨): 바이마 알림 메일 폴링 (Gmail API + 라벨 필터) → `cs_threads` 자동 생성 — 죽어있는 CS cron을 이 흐름으로 교체
- AI 초안: 템플릿 50선을 `cs_templates` 테이블에 시드 → Claude가 문의 분류 후 **가장 가까운 템플릿 기반**으로 초안 (자유 생성보다 안전)
- 사장님 화면: 초안 확인 → 원클릭 복사 (바이마 메시지창 자동입력은 확장으로 추후)

## 9. 모듈 7 — AI 직원 / 대표 비서

- **AI 직원** = 파이프라인 그 자체 (수집→검증→준비→모니터링, 전부 자동)
- **대표 비서** = `/today` 개편: "오늘 결정할 것"만 모아서 브리핑
  - 아침 브리핑: 신규 후보 N개 (근거 요약) / 가격 방어 제안 M건 / CS 초안 K건 / 주문 처리 L건
  - 모든 항목이 원클릭 결정 (승인/거부/보류)
  - 주간 리포트: 소싱 적중률(셀렉률·판매전환), 패스 사유 트렌드, 마진 추이
- 분업화 가이드 반영: 대표 화면은 "결정"만, 정보 탐색이 필요 없는 구조
